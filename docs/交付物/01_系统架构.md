## 一、总体架构

系统采用**管道-过滤器**架构，以 `main.py` 为调度中枢，将四个数据库的并发扫描、六种数据形态的智能路由、20 种敏感类型的识别与校验、流式去重写出等功能分层解耦。

```
╔══════════════════════════════════════════════════════════════════╗
║                    main.py  扫描总调度                           ║
║  • ThreadPoolExecutor(DB_WORKERS=2)  四库并发                    ║
║  • 时间预算：总 30min，单表熔断 TABLE_TIMEOUT=540s               ║
╚══════════════════╤═══════════════════════════════════════════════╝
                   │
       ┌───────────┴────────────┐
       ▼                        ▼
  db_connector.py          csv_writer.py
  ─────────────            ─────────────
  • MySQL / PostgreSQL     • 七元组去重键
  • 流式游标 server-side   • UTF-8 无BOM
  • 整数主键嗅探           • 实时 flush
  • 自动识别 BLOB/BYTEA    • RFC 4180 引号

                   │
                   ▼
           dispatcher.py   ← 形态嗅探路由
                   │
    ┌──────┬───────┼────────┬──────────┐
    ▼      ▼       ▼        ▼          ▼
struct  encoded  unstruct  blob    dbobject
  │       │        │        │          │
正则   递归解码  正则+     OCR+     SQL扫描
+校验  5轮链    姓氏锚点  文档解析  存储过程
                           │          视图
                    ocr_client.py
                    ocr_worker.py
                    (子进程隔离)
```

## 二、各模块详细说明

### 2.1 调度层（`main.py`）

| 职责 | 实现 |
|------|------|
| 四库并发 | `ThreadPoolExecutor(max_workers=2)` 同时扫描 MySQL + PostgreSQL |
| BLOB 专用并发 | BLOB 表内行级并发 `ThreadPoolExecutor(max_workers=OCR_POOL_SIZE=2)` |
| 时间预算 | 全局 30min；单表 `TABLE_TIMEOUT_SECONDS=540s` 熔断；BLOB 行级 OCR 超时 25s |
| 数据库对象 | `scan_db_objects()` 扫存储过程 + 视图，在常规表扫描后追加 |
| 噪声过滤 | `N/A_数字` 格式行跳过，空值跳过 |

### 2.2 连接层（`core/db_connector.py`）

- **严格使用 `127.0.0.1`**，不使用 `localhost`（避免 Unix Socket 路径问题）
- 服务端流式游标（`SSCursor` / `server_side_cursor`），避免一次性加载大表
- 自动嗅探整数主键列；无主键时用行偏移量作为 `record_id`
- BLOB/BYTEA 字段检测：类型名含 `blob`/`bytea`/`binary`/`varbinary` 的列路由到 BLOB 扫描器

### 2.3 分发路由（`core/dispatcher.py`）

按以下优先级顺序嗅探，命中即路由：

```
1. isinstance(value, bytes)           → blob 扫描器
2. PostgreSQL \x 十六进制前缀         → blob 扫描器
3. 字段名含 password/token/secret     → PASSWORD_OR_SECRET（结构化）
4. value 以 { / [ 开头               → JSON 递归解析
5. value 以 < 开头                   → XML 解析
6. 具编码特征（含字母的 Base64/Hex）  → encoded 扫描器
7. 字段名强语义提示（real_name 等）   → structured（带字段提示）
8. 长文本（len > 阈值）              → unstructured_text
9. 默认                              → structured
```

### 2.4 识别层——六种形态处理器

#### ① 结构化（`scanners/structured.py`）

- **字段名语义字典**：`FIELD_NAME_DICT`（20 类 × 多关键词，词边界 `\b` 匹配）
- **全量正则**：对字段值并行执行 20 种 `REGEX_PATTERNS`
- **校验函数**：身份证 GB 11643 校验位、银行卡 Luhn、USCC 字符集 + 非银行卡前缀

#### ② 编码（`scanners/encoded.py`）

- 递归解码链，最多 5 轮，每轮尝试：URL → Base64 → Hex → Unicode 转义
- 每轮产物再尝试 JSON/XML 解析（编码套半结构化场景）
- 四重熔断：深度 / 长度 / 循环自指检测 / 膨胀比 > 100

#### ③ 半结构化（`scanners/structured.py` JSON/XML 分支）

- `json.loads()` 递归展开，所有叶子字符串值逐一送 `extract_sensitive_from_value()`
- `xml.etree.ElementTree` 解析 XML，提取所有文本节点与属性值

#### ④ 非结构化文本（`scanners/unstructured.py`）

- 直接调用 `extract_sensitive_from_value()`（正则全套 + 中文姓名 NFA 遍历）
- 中文姓名：百家姓锚点 + 复姓词典 + 多层过滤（黑名单、行政区防卫、动词后缀过滤）
- 可选：PaddleNLP UIE 本地 NER 模型补充召回（`UIE_ENABLED=1` 开启）

#### ⑤ 二进制 BLOB（`scanners/blob.py` + `ocr_client/worker.py`）

```
BLOB 字节
    ↓ _normalize_to_bytes()       (bytes / memoryview / hex串 统一化)
    ↓ _sniff_file_type()          (magic bytes 白名单识别)
    ↓ 路由
    ├─ PDF  → PyMuPDF 数字文本 / 扫描版逐页 OCR
    ├─ DOCX → python-docx 提取段落 + 表格
    ├─ XLSX → openpyxl 遍历单元格
    ├─ image(JPEG/PNG/GIF/BMP/TIFF/WEBP) → PaddleOCR
    └─ unknown → 字节级文本兜底（UTF-8/GBK 解码）

OCR 后处理：
    • _preprocess_ocr_text()      (数字空格合并, MR/YB 归一, 汉字间空格, 地址跨行)
    • 多路径扫描：原文/数字纠错/汉字纠错/双修/去空格  共5路并行
    • _scan_digit_windows()       (11-20位数字串: 手机号/身份证/银行卡 兜底)
    • _enumerate_digit_fixes()    (1-edit BFS 枚举: OCR digit→digit 混淆修复)
```

**OCR 子进程隔离**：PaddleOCR 运行于独立子进程池，防止 C++ 层段错误导致主进程崩溃；超时 25s 即取消，子进程连续失败后自动重建。

#### ⑥ 数据库对象（`scanners/dbobject.py`）

- 查询 MySQL `information_schema.ROUTINES/VIEWS`、PostgreSQL `pg_proc/information_schema.views`
- 对存储过程/函数/视图的 SQL 定义体执行全套正则扫描
- 输出中 `table_name` 填 `__stored_procedure__` / `__view__`，`record_id` 填 `—`

### 2.5 核心模式库（`core/patterns.py`）

| 组件 | 说明 |
|------|------|
| `REGEX_PATTERNS` | 20 种敏感类型的正则表达式，含边界断言防重叠 |
| `FIELD_NAME_DICT` | 字段名语义字典，词边界精确匹配 |
| `COMMON_SURNAMES` / `COMPOUND_SURNAMES` | 百家姓单姓 + 复姓词典 |
| `NAME_BLACKLIST` | 非名字高频词黑名单（系统词、地名、药品词等） |
| `_NAME_FOLLOW_VERB_BIGRAMS` | 姓名后缀动词二元组（过滤"方违约"类误报） |
| `validate_id_card()` | GB 11643 校验位算法 |
| `validate_luhn()` | Luhn 算法（银行卡） |
| `validate_uscc()` | USCC 字符集 + 银行卡前缀排除 |
| `is_valid_address()` | 地址有效性：行政区划字 + 街道后缀 + 长度 + 禁用词 |
| `extract_sensitive_from_value()` | 统一扫描入口，返回 `[(type, value)]` |

### 2.6 脱敏工具（`core/masking.py`）

- 20 种敏感类型各有专属脱敏函数（见 §02_脱敏方案.md）
- `mask_value(type, value)` 统一调度
- `batch_mask_csv(input, output)` 批量生成脱敏前后对比报告

### 2.7 输出层（`core/csv_writer.py`）

- 七元组 `(db_type, db_name, table_name, field_name, record_id, sensitive_type, extracted_value)` 去重
- 每条记录实时 `flush`，防崩溃丢失
- UTF-8 无 BOM，LF 换行，RFC 4180 最小化引号

## 三、数据流向图

```
MySQL/PostgreSQL
      │
      ▼ 流式游标（server-side cursor）
  行记录 (pk, field_name, value, field_type)
      │
      ▼ dispatcher.py 嗅探
 ┌────┴────────────────────────────────────┐
 │  形态判断：bytes? JSON? Base64? 长文本?  │
 └────┬────────────────────────────────────┘
      │
      ├─ structured  → REGEX × 20 + 校验 → [(type, val)]
      ├─ semi_struct → JSON/XML 递归 → [(type, val)]
      ├─ encoded     → 5轮解码链 → [(type, val)]
      ├─ unstructured→ REGEX + 姓名NFA → [(type, val)]
      ├─ blob        → OCR/文档 + 多路径 → [(type, val)]
      └─ dbobject    → SQL体扫描 → [(type, val)]
           │
           ▼ csv_writer（七元组去重 + flush）
      output/upload.csv
           │
           ▼ core/masking.py（可选脱敏演示）
      output/masked_report.csv
```

## 四、性能设计

| 策略 | 具体措施 |
|------|---------|
| **并发** | 4库 × 2线程并发扫描；BLOB 行级 2 线程并发 OCR |
| **熔断** | 单表 540s 超时跳过；单行 OCR 25s 超时取消 |
| **流式** | Server-side cursor，避免大表一次性加载内存 |
| **子进程隔离** | OCR 在独立进程池运行，崩溃不影响主进程 |
| **短路** | 字段值 `N/A_数字` 等噪声直接跳过；字段名黑名单不送 NLP |
| **去重** | 七元组 `set` 去重，内存内秒级查重 |
| **延迟加载** | PyMuPDF / python-docx / openpyxl 首次用到时才 `import` |

**目标指标**（赛题硬性约束）：

| 指标 | 目标 |
|------|------|
| 全量扫描时间 | ≤ 30 分钟 |
| 内存峰值 | ≤ 500 MB |
| CPU 占用 | 尽可能低（影响性能得分） |

## 五、技术选型说明

| 需求 | 选型 | 理由 |
|------|------|------|
| 中文 OCR | PaddleOCR ch_PP-OCRv4（CPU 推理） | 离线，中文识别率高，赛题指定 |
| PDF 文本抽取 | PyMuPDF (fitz) | 速度快，支持数字版直接抽文本 |
| Word 解析 | python-docx | 无需 Office 授权，纯 Python |
| Excel 解析 | openpyxl | 纯 Python，read_only 模式省内存 |
| NER（可选） | PaddleNLP UIE-base | 离线，精度高，与 PaddleOCR 同框架 |
| 正则引擎 | Python `re`（CPython C 扩展） | 无需额外依赖，性能足够 |
| 数据库驱动 | `pymysql`（MySQL）/ `psycopg2`（PostgreSQL） | 成熟稳定，支持 server-side cursor |

## 六、合规与安全设计

- **严格离线**：无任何网络请求，所有模型预置于本地
- **数据不出机**：扫描结果仅写入本地 `output/upload.csv`
- **日志脱敏**：`scan_error.log` 只记录连接错误和超时警告，不记录敏感值明文
- **分级分类**：对齐 GB/T 35273-2020《个人信息安全规范》和《数据安全法》四级框架
