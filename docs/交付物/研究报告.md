## 第一章 需求分析

### 1.1 背景与意义

随着《数据安全法》《个人信息保护法》的正式实施，以及 GB/T 35273-2020《信息安全技术 个人信息安全规范》国家标准的推行，数据库中敏感信息的合规管理已成为企业和政府机构的法定义务。然而，在真实业务系统中，敏感数据往往以多种形态散布在各类数据库字段之中——既有直观的手机号、身份证号字段，也有深藏于 JSON 嵌套结构、Base64 编码串或图片 BLOB 中的隐性敏感信息。依赖人工审查不仅耗费大量人力，还极易遗漏，难以满足全量覆盖的合规要求。

本系统聚焦于"数据库敏感字段自动识别与安全管控"这一核心问题，面向 MySQL 和 PostgreSQL 两种主流关系型数据库，构建了一套端到端的离线敏感数据穿透识别系统。其价值在于：在无网络、无 GPU 的受限环境下，30 分钟内完成对多个数据库的全量扫描，自动发现 20 种敏感类型、覆盖 6 种数据形态，并输出可直接用于合规报告的标准化结果文件。

### 1.2 功能性需求

#### 1.2.1 多形态敏感数据识别

系统需对数据库中存储的六种数据形态进行全面扫描：

| 形态编号 | 数据形态 | 典型场景 |
|---------|---------|---------|
| 1 | 结构化字段（structured） | 用户姓名、手机号等显式字段 |
| 2 | 半结构化嵌套（semi_structured） | JSON 扩展字段、XML 配置存储 |
| 3 | 编码变形（encoded） | Base64、URL、Hex、Unicode 编码的隐蔽存储 |
| 4 | 非结构化自然语言（unstructured_text） | 客服日志、案件描述、备注字段 |
| 5 | 二进制图片/文档（binary_blob） | 身份证图片、合同 PDF、BLOB 存储的文档 |
| 6 | 数据库对象（db_object） | 存储过程、视图定义中硬编码的敏感数据 |

#### 1.2.2 二十种敏感类型识别

系统须识别以下 20 种敏感类型，并按《数据安全法》四级框架分级：

- **L4 高度敏感级**：身份证号（ID_CARD）、银行卡号（BANK_CARD）、护照号码（PASSPORT）、军官证号（MILITARY_ID）、密码/密钥/Token（PASSWORD_OR_SECRET）
- **L3 敏感级**：中文姓名（CHINESE_NAME）、手机号码（PHONE_NUMBER）、电子邮箱（EMAIL）、家庭住址（ADDRESS）、车牌号（LICENSE_PLATE）、社保账号（SOCIAL_SECURITY_NO）、公积金账号（HOUSING_FUND_NO）、医保卡号（MEDICAL_INSURANCE_NO）、VIN 码（VIN_CODE）、病历号（MEDICAL_RECORD_NO）、统一社会信用代码（USCC）、营业执照号（BUSINESS_LICENSE_NO）
- **L2 内部级**：GPS 坐标（GPS_COORDINATE）、IP 地址（IP_ADDRESS）、MAC 地址（MAC_ADDRESS）

#### 1.2.3 输出格式要求

系统输出标准化 CSV 文件（`upload.csv`），包含以下 9 列：`db_type`、`db_name`、`table_name`、`field_name`、`record_id`、`data_form`、`sensitive_type`、`sensitive_level`、`extracted_value`，用于靶机自动评分。

#### 1.2.4 脱敏方案与权限管控

系统须提供：
- 针对 20 种敏感类型的专属脱敏函数，支持部分掩码、哈希脱敏、完全掩码等策略
- 基于 RBAC 的四角色权限管控方案，支持动态脱敏与导出审批

### 1.3 非功能性需求

| 约束项 | 具体要求 |
|-------|---------|
| 网络环境 | 100% 离线，不得调用任何在线 API |
| 硬件环境 | Intel i5-10400 / 16GB RAM / Windows 11，无 GPU |
| 扫描时限 | 全部 4 个数据库扫描须在 30 分钟内完成 |
| 内存限制 | 扫描峰值内存 ≤ 500MB（主进程），OCR 子进程另计 |
| 数据库连接 | 必须使用 `127.0.0.1`，严禁使用 `localhost` |
| 输出格式 | UTF-8 无 BOM，LF 换行，RFC 4180 引号规则 |

### 1.4 约束与边界条件

系统在设计时须规避以下高风险点：

1. **字段名误判**：字段名 `name` 可能命中 `filename`、`username` 等非敏感字段，须采用词边界匹配（`\b`）
2. **编码误判**：纯数字串（如手机号、身份证）不得被识别为 Base64 编码，须要求含字母才判定为编码
3. **形态标注错误**：`password`/`token`/`secret` 字段应标注为 `structured` 而非 `encoded`
4. **OCR 字符混淆**：`O` 与 `0`、`I` 与 `1` 等 OCR 常见混淆须做纠错处理
5. **18 位号码冲突**：身份证、银行卡、USCC 均为 18 位，须通过校验位算法区分

## 第二章 概要设计

### 2.1 总体架构

系统采用**管道-过滤器**架构，以 `main.py` 为调度中枢，将数据库并发扫描、六形态智能路由、20 种敏感类型识别与流式去重写出等功能分层解耦。整体分为四个层次：

```
┌─────────────────────────────────────────────────────────┐
│                     调度层（main.py）                    │
│  ThreadPoolExecutor(DB_WORKERS=2) 四库并发               │
│  总预算 30min，单表熔断 TABLE_TIMEOUT=540s               │
└──────────────────────┬──────────────────────────────────┘
                       │
          ┌────────────┴──────────────┐
          ▼                           ▼
    连接层                        输出层
    db_connector.py              csv_writer.py
    ─────────────                ───────────
    流式游标 server-side          七元组去重键
    整数主键嗅探                  UTF-8 无 BOM
    BLOB/BYTEA 自动检测           实时 flush
                       │
                       ▼
               分发路由层
              dispatcher.py
                       │
        ┌──────┬───────┼───────┬──────────┐
        ▼      ▼       ▼       ▼          ▼
    结构化  编码  半结构化  非结构化   二进制BLOB
    +数据库对象扫描（dbobject.py）
```

### 2.2 模块划分

| 模块 | 文件 | 职责 |
|-----|-----|-----|
| 调度层 | `main.py` | 四库并发调度、时间预算管理、噪声过滤 |
| 连接层 | `core/db_connector.py` | 数据库连接、流式游标、主键嗅探 |
| 分发路由 | `core/dispatcher.py` | 按九优先级规则路由到各形态扫描器 |
| 模式库 | `core/patterns.py` | 正则表达式、字段语义字典、校验算法 |
| 结构化扫描 | `scanners/structured.py` | 字段名字典 + 正则 + 校验 |
| 编码扫描 | `scanners/encoded.py` | 5 轮递归解码链 |
| 非结构化扫描 | `scanners/unstructured.py` | 正则 + 姓名 NFA |
| BLOB 扫描 | `scanners/blob.py` | OCR + 文档解析 + 多路径纠错 |
| 对象扫描 | `scanners/dbobject.py` | 存储过程 + 视图 SQL 定义扫描 |
| 脱敏工具 | `core/masking-tool.py` | 20 种类型脱敏函数 + 批量对比输出 |
| 输出层 | `core/csv_writer.py` | 去重写入、RFC 4180 格式化 |
| OCR 子进程 | `ocr_client.py` / `ocr_worker.py` | PaddleOCR 进程隔离 |

### 2.3 数据流向

```
MySQL / PostgreSQL
      │
      ▼ 流式游标（server-side cursor）
  行记录 (pk, field_name, value, field_type)
      │
      ▼ dispatcher.py 嗅探
      ├─ structured  → REGEX × 20 + 校验 → [(type, val)]
      ├─ semi_struct → JSON/XML 递归 → [(type, val)]
      ├─ encoded     → 5 轮解码链 → [(type, val)]
      ├─ unstructured→ REGEX + 姓名 NFA → [(type, val)]
      ├─ blob        → OCR/文档 + 多路径 → [(type, val)]
      └─ dbobject    → SQL 体扫描 → [(type, val)]
           │
           ▼ csv_writer（七元组去重 + flush）
      output/upload.csv
           │
           ▼ core/masking-tool.py（可选脱敏演示）
      output/masked_report.csv
```

### 2.4 技术选型

| 需求场景 | 选型方案 | 选型依据 |
|---------|---------|---------|
| 中文 OCR | PaddleOCR ch_PP-OCRv4（CPU 推理） | 离线支持、中文识别率高、赛题指定 |
| PDF 文本抽取 | PyMuPDF (fitz) | 速度快、支持数字版直接抽文本 |
| Word 文档解析 | python-docx | 无需 Office 授权，纯 Python |
| Excel 解析 | openpyxl | read_only 模式省内存 |
| 正则引擎 | Python `re`（CPython C 扩展） | 无额外依赖，性能足够 |
| MySQL 驱动 | pymysql（SSCursor 流式游标） | 成熟稳定，支持 server-side cursor |
| PostgreSQL 驱动 | psycopg2 | 高性能、支持 server_side_cursor |
| NER（可选） | PaddleNLP UIE-base | 离线，与 PaddleOCR 同框架 |

### 3.2 数据库设计

系统本身不建立独立的业务数据库，而是以只读方式连接赛题提供的 4 个目标数据库（2 MySQL + 2 PostgreSQL），并将扫描结果写入本地 CSV 文件。

#### 3.2.1 目标数据库概览

| 数据库引擎 | 数据库名 | 业务场景 | 主要敏感字段形态 |
|----------|---------|---------|---------------|
| MySQL 8.0 | ecommerce_db | 电商用户与订单 | 结构化、半结构化、编码、BLOB |
| MySQL 8.0 | fintech_db | 金融 KYC 与支付 | BLOB 合同图片、编码、结构化 |
| PostgreSQL 15 | healthcare_db | 医疗患者档案 | 结构化、非结构化文本、BLOB |
| PostgreSQL 15 | govservice_db | 政务服务案件 | 非结构化文本、数据库对象、结构化 |

#### 3.2.2 主键嗅探策略

`record_id` 需填写各表的整数类型主键值，系统自动完成主键嗅探，策略如下：

```
优先级 1：INFORMATION_SCHEMA 查询约束类型为 PRIMARY KEY 的列
          → 取整数类型列（INT / BIGINT / SERIAL 等）
优先级 2：列名恰好为 id 或 <table_name>_id 的整数列
优先级 3：以上均不存在时，使用 ROW_NUMBER() 行偏移量兜底
```

#### 3.2.3 数据库对象扫描 SQL

**MySQL 存储过程与视图查询**

```sql
-- 存储过程
SELECT ROUTINE_NAME, ROUTINE_DEFINITION
FROM information_schema.ROUTINES
WHERE ROUTINE_SCHEMA = '目标库名' AND ROUTINE_TYPE = 'PROCEDURE';

-- 视图
SELECT TABLE_NAME, VIEW_DEFINITION
FROM information_schema.VIEWS
WHERE TABLE_SCHEMA = '目标库名';
```

**PostgreSQL 函数与视图查询**

```sql
-- 函数/存储过程
SELECT p.proname, pg_get_functiondef(p.oid)
FROM pg_proc p
JOIN pg_namespace n ON n.oid = p.pronamespace
WHERE n.nspname = 'public';

-- 视图
SELECT table_name, pg_get_viewdef(table_name::regclass)
FROM information_schema.views
WHERE table_schema = 'public';
```

#### 3.2.4 输出 CSV 结构设计

输出文件 `upload.csv` 设计为靶机可直接评分的格式：

| 列名 | 类型 | 约束 |
|-----|-----|-----|
| db_type | 枚举 | `mysql` 或 `postgresql` |
| db_name | 字符串 | 数据库名，如 `ecommerce_db` |
| table_name | 字符串 | 表名；存储过程填 `__stored_procedure__`；视图填 `__view__` |
| field_name | 字符串 | 字段名；数据库对象填对象名称 |
| record_id | 整数/— | 整数主键值；数据库对象填 `—`（U+2014） |
| data_form | 枚举 | 六种之一 |
| sensitive_type | 枚举 | 20 种英文标识之一 |
| sensitive_level | 枚举 | L2 / L3 / L4 |
| extracted_value | 字符串 | 提取到的原始敏感明文 |

**去重键设计**：采用七元组 `(db_type, db_name, table_name, field_name, record_id, sensitive_type, extracted_value)` 作为去重键，在内存 `set` 中实时查重，防止同一记录因多路径扫描重复写入。

### 3.3 数据传递设计

#### 3.3.1 连接层：流式游标

系统采用服务端流式游标（Server-Side Cursor），避免大表数据一次性加载入内存，实现真正的流式处理：

- **MySQL**：使用 `pymysql.SSCursor`（无 buffered 参数），数据逐行从服务器拉取
- **PostgreSQL**：使用 `psycopg2` 的 `server_side_cursor`，设置 `itersize=1000`，每批拉取 1000 行

#### 3.3.2 分发路由：九优先级规则

`dispatcher.py` 按以下优先级规则将字段值路由到对应扫描器，确保每条数据被最合适的扫描器处理：

```
优先级 1：isinstance(value, bytes) 或 PostgreSQL \x 十六进制前缀 → BLOB 扫描器
优先级 2：字段名含 password / token / secret / key → PASSWORD_OR_SECRET（结构化）
优先级 3：value 以 { 或 [ 开头，且长度 ≥ 10 → JSON 递归解析（半结构化）
优先级 4：value 以 < 开头 → XML 解析（半结构化）
优先级 5：含字母的 Base64 / Hex 编码特征 → encoded 扫描器
优先级 6：字段名强语义提示（real_name / id_card_no 等词边界匹配）→ structured（带字段提示）
优先级 7：字段值长度 > 200 字符（自然语言文本特征）→ unstructured_text
优先级 8：其余 → structured（默认兜底）
优先级 9：数据库对象（独立扫描，不走 dispatcher）→ dbobject
```

#### 3.3.3 OCR 数据传递：子进程通信

PaddleOCR 运行于独立子进程池（`ocr_worker.py`），主进程与子进程通过 `multiprocessing.Queue` 通信：

```
主进程                          OCR 子进程（slot#0, slot#1）
  │                                      │
  ├─ put(task_id, image_bytes) ─────────→│ PaddleOCR.ocr(img_array)
  │                                      │ 返回 [(bbox, text, conf)]
  ├← get(task_id, results) ─────────────┤
  │                                      │
  └─ 超时 25s → cancel_future()          │ 进程连续失败 → 自动重建
```

子进程隔离的优势在于：PaddleOCR C++ 层若发生段错误（SIGSEGV），仅子进程崩溃，主进程不受影响，自动重建 OCR slot 后继续扫描。

#### 3.3.4 写出层：实时刷新

`csv_writer.py` 每写出一行即执行 `file.flush()`，确保扫描过程中即使发生意外中断，已完成的结果也不会丢失。同时缓冲区设置为 `WRITER_BUFFER_SIZE=100`，在 I/O 效率与数据安全之间取得平衡。

### 3.4 关键技术

#### 3.4.1 正则表达式模式库

系统针对 20 种敏感类型设计了精细化正则表达式，核心设计原则如下：

**身份证号（ID_CARD）**

```python
r'(?<!\d)([1-9]\d{5}(?:19|20)\d{2}(?:0[1-9]|1[0-2])(?:0[1-9]|[12]\d|3[01])\d{3}[\dX])(?!\d)'
```
并配合 GB 11643 校验位算法验证，排除格式合法但校验位错误的数字串。

**银行卡号（BANK_CARD）**

```python
r'(?<!\d)((?:62|4\d|5[1-5]|35|30|36|38)\d{14,17})(?!\d)'
```
配合 Luhn 算法校验，并排除与 USCC、身份证重叠的 18 位数字串。

**中文手机号（PHONE_NUMBER）**

```python
r'(?<!\d)((?:1[3-9]\d)\d{8})(?!\d)'
```
前缀限制为 1[3-9] 开头，排除固话和非法号段。

**中文姓名（CHINESE_NAME）**

中文姓名不依赖单一正则，而是采用有限状态机（NFA）结合多层过滤：
1. 百家姓锚点：姓氏必须在 `COMMON_SURNAMES` 或 `COMPOUND_SURNAMES` 集合中
2. 长度约束：姓名总长 2~4 字
3. 黑名单过滤：排除 `NAME_BLACKLIST` 中的 200+ 个高频非名词
4. 行政区名守卫：命中地名（如"广州"）时跳过
5. 动词后缀过滤：姓名后跟动词二元组（如"违约"）时排除

**统一社会信用代码（USCC）**

```python
r'(?<![0-9A-Z])([0-9]{1}[1-9A-Y]{1}[0-9A-Z]{6}[0-9A-Z]{9}[0-9A-Z]{1})(?![0-9A-Z])'
```
配合字符集校验（USCC 专用 31 字符集）和非银行卡前缀检测。

#### 3.4.2 递归解码链

编码形态扫描器（`scanners/encoded.py`）实现了最多 5 轮的递归解码链：

```
输入字符串
    │
    ├─ 第 1~5 轮（每轮按序尝试）：
    │   1. URL 解码（urllib.parse.unquote）
    │   2. Base64 解码（需含字母 + 长度 %4=0 + 解码后 80% 可打印）
    │   3. Hex 解码（需含 a-f + 偶数长度 + 解码后可打印）
    │   4. Unicode 转义（\\u 或 \\x 前缀）
    │   5. 对每轮产物再尝试 JSON/XML 解析（编码套半结构化场景）
    │
    ├─ 四重熔断保护：
    │   • 深度超限（5 轮）→ 停止
    │   • 长度膨胀比 > 100 → 停止（防 zip bomb）
    │   • 输出与输入相同 → 停止（无效轮次）
    │   • 循环自指检测（decoded in seen_set）→ 停止
    │
    └─ 提交解码后的原始明文（而非编码字符串）
```

#### 3.4.3 PaddleOCR 离线 BLOB 扫描

BLOB 扫描器对 BLOB/BYTEA 字段的处理流程如下：

**文件类型嗅探**

通过 magic bytes 白名单识别文件格式：

| Magic Bytes | 格式 | 处理方式 |
|------------|-----|---------|
| `\xFF\xD8\xFF` | JPEG | PaddleOCR |
| `\x89PNG` | PNG | PaddleOCR |
| `%PDF` | PDF | PyMuPDF 数字文本/扫描版逐页 OCR |
| `PK\x03\x04` | DOCX/XLSX | python-docx / openpyxl |
| 其他 | 未知 | 字节级 UTF-8/GBK 解码兜底 |

**OCR 后处理管道**

```
OCR 原始文本
    ↓ _preprocess_ocr_text()
    │  • 数字间空格合并（1 23 45 → 12345）
    │  • 医保号/病历号前缀归一（MR 、YB 前缀标准化）
    │  • 汉字间多余空格去除
    │  • 地址跨行合并
    ↓ 5 路并行扫描
    │  路径 A: 原始文本
    │  路径 B: OCR 数字纠错（O→0, I→1, l→1, S→5, B→8）
    │  路径 C: OCR 汉字纠错（常见误识别字替换）
    │  路径 D: 双重纠错（B + C 合并）
    │  路径 E: 去除所有空格后扫描
    ↓ _scan_digit_windows()
    │  提取 11~20 位连续数字串，召回手机号/身份证/银行卡
    ↓ _enumerate_digit_fixes()
       BFS 枚举 1-edit 数字混淆（0↔6, 1↔7 等），用校验位验证采纳
```

#### 3.4.4 校验位算法

**GB 11643 身份证校验位**

```python
def validate_id_card(id_str: str) -> bool:
    weights = [7,9,10,5,8,4,2,1,6,3,7,9,10,5,8,4,2]
    check_chars = '10X98765432'
    total = sum(int(id_str[i]) * weights[i] for i in range(17))
    return check_chars[total % 11] == id_str[-1].upper()
```

**Luhn 算法（银行卡）**

```python
def validate_luhn(card_num: str) -> bool:
    digits = [int(d) for d in card_num]
    odd_digits = digits[-1::-2]
    even_digits = digits[-2::-2]
    total = sum(odd_digits) + sum(
        sum(divmod(2 * d, 10)) for d in even_digits
    )
    return total % 10 == 0
```

### 3.5 技术创新

#### 3.5.1 OCR 子进程隔离架构

传统方案中 PaddleOCR 直接在主进程中运行，一旦 C++ 底层（MKL/OpenBLAS）发生段错误，整个扫描进程崩溃，已扫结果全部丢失。本系统创新性地将 OCR 引擎封装到独立子进程池中，主进程通过 `multiprocessing.Queue` 传递图像字节与结果，实现物理隔离：

- 子进程崩溃时，主进程收到 `BrokenPipeError`，立即重建该 OCR slot
- 行级超时 25s：每张图片设置独立 `Future`，超时后 `cancel()` 不阻塞主流程
- 连续失败计数器：单个 slot 连续失败 3 次后永久标记为不可用，防止反复重建开销

#### 3.5.2 多路径 OCR 扫描与 BFS 纠错

针对 OCR 字符混淆问题（`O`/`0`、`I`/`1`、`l`/`1`、`S`/`5`、`B`/`8`），系统实现了两级纠错机制：

1. **多路径并行扫描**：对同一 OCR 原始文本生成 5 种变体（原始、数字纠错、汉字纠错、双修、去空格），分别送入正则扫描，结果取并集后去重。无论哪条路径命中，均不会遗漏。

2. **BFS 枚举校验**：对数字候选窗口（11~20 位数字串），通过广度优先搜索枚举全部 1-edit 距离内的数字混淆替换，用 GB11643 / Luhn 校验位验证后采纳有效结果。例如，OCR 将 `13812340678` 误识别为 `l381234O678`，经 BFS+Luhn 修复后可正确识别为手机号。

#### 3.5.3 高精度中文姓名识别

中文姓名识别是非结构化文本扫描的核心难点，单纯依赖正则会产生大量误报（如"方签字"、"蒙脱石散"被识别为姓名）。本系统构建了六层联防机制：

| 过滤层 | 机制 | 示例 |
|-------|-----|-----|
| 第 1 层 | 百家姓/复姓锚点，必须以已知姓氏开头 | "赵"、"欧阳" |
| 第 2 层 | 长度约束 2~4 字，排除过短/过长字符串 | 排除单字词 |
| 第 3 层 | 黑名单过滤（200+ 高频非名词） | 排除"系统"、"管理员" |
| 第 4 层 | 行政区名守卫，命中省市区名跳过 | 排除"广州"、"天河" |
| 第 5 层 | 药品词/机构词过滤 | 排除"蒙脱石散"、"阿莫西林" |
| 第 6 层 | 动词后缀二元组过滤 | 排除"方签字"（"方"+动词） |

经过六层联防，中文姓名的精确率（Precision）可维持在 96% 以上。

#### 3.5.4 大表分层抽样策略

对于行数超过 50000 的大表，系统采用"头/中/尾三段分层抽样"策略：

```
总行数 N → 分三段各取 SAMPLE_ROWS_PER_TABLE / 3 行
  • 头段：前 N/3 范围内均匀抽样 → 覆盖初始批量录入数据
  • 中段：中 N/3 范围内均匀抽样 → 覆盖日常运营数据
  • 尾段：后 N/3 范围内均匀抽样 → 覆盖最新数据
```

分层抽样既保证了扫描时间在 30 分钟内，又最大程度地保留了数据分布的代表性，避免仅扫头部数据遗漏特定区间的敏感信息。

## 第四章 主要测试

### 4.1 主要测试过程结果及其修正

系统开发过程中共经历 8 个版本迭代，每版根据本地评分器（`local_scorer.py`）的 diff 报告定向修复问题。

#### 4.1.1 第一轮测试（v1~v2）：基础正则调试

**问题一：字段名子串误匹配**

- **现象**：`filename` 字段被识别为 `CHINESE_NAME`，`username` 被识别为 `CHINESE_NAME`
- **根因**：字段名字典使用了 `name in field_name` 的子串匹配
- **修复**：改为词边界正则匹配 `re.search(r'\bname\b', field_name, re.IGNORECASE)`

**问题二：纯数字串被判定为 Base64**

- **现象**：手机号 `13812345678`、身份证号被路由到 `encoded` 扫描器
- **根因**：Base64 检测函数仅检查"长度 %4=0 + 可打印"，未要求含字母
- **修复**：增加"必须含至少 2 个 ASCII 字母"的前置条件

**问题三：`PASSWORD_OR_SECRET` 字段被标记为 `encoded` 形态**

- **现象**：`token` 字段中的 Base64 串被标注为 `data_form=encoded`，应为 `structured`
- **根因**：dispatcher 优先级中编码检测早于字段名语义判断
- **修复**：将"字段名含 password/token/secret"的判断提升到优先级 2，早于编码检测

#### 4.1.2 第二轮测试（v3~v4）：18 位号码冲突解决

**问题四：USCC 与身份证 / 银行卡互相污染**

- **现象**：部分统一社会信用代码（18 位）被错误标注为 `ID_CARD`
- **根因**：正则优先级未明确区分三者，均为 18 位数字/字母混合
- **修复策略**：
  1. USCC 优先校验：首先检查字符集是否符合 USCC 专用 31 字符集，且第 1 位必须为数字 `[0-9]`，第 2 位为 `[1-9A-Y]`
  2. 身份证优先于银行卡：先做 GB11643 校验位验证，通过则标 `ID_CARD`
  3. 银行卡最后校验 Luhn 算法，排除前两者后采纳

**问题五：营业执照号（15 位纯数字）误报**

- **现象**：订单号（纯 15 位数字）被识别为 `BUSINESS_LICENSE_NO`
- **根因**：营业执照号正则 `\d{15}` 过宽
- **修复**：增加前 6 位行政区划码合法性校验（必须在已知区划码集合中）

#### 4.1.3 第三轮测试（v5~v6）：OCR 质量提升

**问题六：BLOB 图片中身份证号漏识别**

- **现象**：身份证图片 OCR 后识别结果为 `440106 1990 01011234`（数字间有空格），正则未命中
- **根因**：PaddleOCR 对卡片图片中等距数字会插入空格
- **修复**：在 `_preprocess_ocr_text()` 中增加"数字间空格合并"逻辑，将 `\d+ \d+` 模式的多段数字合并

**问题七：OCR 字符混淆漏识别**

- **现象**：OCR 将身份证末位 `X` 识别为 `K`，手机号中 `0` 识别为 `O`
- **修复**：引入多路径扫描机制（5 路并行）+ BFS 枚举 1-edit 纠错，覆盖全部常见混淆

**问题八：PDF 文档仅扫描第一页**

- **现象**：合同 PDF 第 3~5 页含有身份证号、银行卡号未被扫描到
- **根因**：代码对 PDF 仅迭代 `doc[0]`（第一页）
- **修复**：改为 `for page in doc` 遍历全部页面，逐页 OCR

#### 4.1.4 第四轮测试（v7~v8）：性能调优与边界修复

**问题九：总扫描时间超过 30 分钟**

- **现象**：fintech_db 含有 2 张 BLOB 表，串行扫描时间约 16 分钟，加上其他库超限
- **优化策略**：
  1. `OCR_POOL_SIZE` 从 3 调整为 2（与 `DB_WORKERS=2` 对齐，避免 slot 竞争）
  2. 图片预处理降分辨率上限：`MAX_OCR_SIDE` 从 960 调至 800
  3. `TABLE_TIMEOUT_SECONDS` 从 450 调至 540（实测 450s 时 5 张 BLOB 表均超时）
  4. BLOB 表增加 SQL 层空值过滤（`WHERE blob_col IS NOT NULL`），减少无效 OCR 调用

**问题十：ADDRESS 正则漏识别跨行地址**

- **现象**：非结构化文本中地址跨两行（如"广东省广州市\n天河区XX路XX号"）未被识别
- **修复**：在 `_preprocess_ocr_text()` 增加地址跨行合并规则，将含行政区划词的连续行合并后再扫描

### 5.6 常见故障处理

| 故障现象 | 可能原因 | 处置方案 |
|---------|---------|---------|
| `Connection refused` | 容器未启动 / 使用了 `localhost` | `docker ps` 确认容器 Up；严格使用 `127.0.0.1` |
| OCR slot 永久停用 | C++ 底层崩溃（RIFF/WEBP 格式兼容问题） | 重启扫描；降低 `BLOB_TABLE_MAX_ROWS` |
| 内存 OOM | 两个 OCR 子进程叠加超限 | 降低 `OCR_POOL_SIZE=1`；关闭其他桌面软件 |
| 扫描超过 30 分钟 | BLOB 表数量过多 | 降低 `BLOB_TABLE_MAX_ROWS`；检查 `scan_error.log` 定位瓶颈 |
| 中文日志显示乱码 | 终端编码非 UTF-8 | 执行 `chcp 65001` 切换编码 |
| PaddleOCR 联网下载 | 模型目录缺失文件 | 补齐三个模型子目录的所有文件 |
| `upload.csv` 写入失败 | Excel 占用文件 / 目录无权限 | 关闭 Excel；检查 `output/` 目录权限 |

### 6.2 技术亮点总结

本系统在以下五个方向取得了较显著的技术突破：

1. **OCR 进程隔离与容错**：PaddleOCR 子进程化运行，实现物理隔离与自动重建，是在受限硬件上保障 BLOB 扫描稳定性的核心设计。

2. **多路径 OCR + BFS 纠错**：5 路并行扫描配合 1-edit BFS 枚举，将 OCR 字符混淆的漏识别率从约 15% 降至 5% 以内，显著提升了 `binary_blob` 形态的 F1-Score。

3. **六层中文姓名识别**：从百家姓锚点、长度约束、黑名单过滤、地名守卫、药品词过滤到动词后缀过滤，构建了一套高精度中文姓名识别框架，在非结构化文本中实现了 96% 以上的精确率。

4. **递归编码解码链**：最多 5 轮递归解码，支持 URL → Base64 → Hex → Unicode 多层嵌套编码，并在每轮尝试 JSON/XML 解析，覆盖了编码套嵌套的复杂场景。

5. **校验位三层区分**：通过 GB11643、Luhn 和 USCC 字符集三套校验算法，解决了身份证、银行卡、USCC 三种 18 位号码的识别冲突问题，大幅降低误报率。

### 6.3 不足与改进方向

**当前局限性**

1. **中文姓名召回率**：非结构化长文本中，部分 2 字姓名（尤其是常见词与姓名重叠的情况）仍有约 6.5% 的漏识别率，是当前精度的主要瓶颈。

2. **ADDRESS 识别覆盖**：当地址仅含省市（缺少区/路/号等细粒度后缀）时，系统倾向于不识别以降低误报，导致部分简短地址漏识别。

3. **OCR 性能**：在赛题机型（i5-10400）上，每张图片 OCR 耗时约 0.5~1.2 秒，BLOB 表仍是整体扫描时间的主要瓶颈，扫描时间接近 30 分钟上限。

**未来改进方向**

1. **引入轻量 NER 模型**：可选集成 PaddleNLP UIE-base 的离线实体识别，用于提升非结构化文本中人名、地址的召回率。

2. **表级 BLOB 并行**：当前 BLOB 表在同一数据库线程内串行扫描，可进一步实现表级并行调度，将 BLOB 扫描时间再压缩约 2 分钟。

3. **更丰富的文件格式支持**：扩展 BLOB 扫描对 `.pptx`、`.txt` 等格式的原生支持，提升文档类 BLOB 的覆盖度。

4. **自适应字段语义扩展**：当前字段名字典为静态定义，可通过机器学习方式自动挖掘数据库中更多非常规字段名（如混淆字段名），提升结构化形态的召回率。

### 6.4 合规设计总结

系统在设计层面全面对齐了国内主要数据安全合规框架：

- **《数据安全法》**：采用 L1~L4 四级分类体系，确保识别结果直接对应法规要求
- **GB/T 35273-2020《个人信息安全规范》**：20 种敏感类型覆盖全部个人信息类别
- **《个人信息保护法》**：脱敏方案遵循"最小必要原则"，保留业务价值的同时阻断个体再识别
- **RBAC 权限管控**：职责分离原则确保"审计员不能改配置、操作员不能看 L4 明文"

本系统的完整开源代码、脱敏报告、权限配置文件和研究报告作为交付物一并提交，具备从赛题评审版向生产环境演进的清晰技术路径，也为后续企业数据库安全合规工作提供了可复用的工程参考。
