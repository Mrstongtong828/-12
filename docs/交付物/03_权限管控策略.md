# 权限管控策略

## 摘要

本文档定义《数据库敏感字段自动识别与安全管控系统》的权限管控总体方案，面向"扫描—查看—导出—审计"全链路给出一套可落地、可审计、可演进的策略体系。方案以 RBAC 为基础，叠加"敏感等级 × 数据库范围 × 操作类型"的多维访问矩阵，并通过动态脱敏、导出审批与 WORM 审计日志三项机制实现最小权限、职责分离与不可抵赖。策略既满足赛题评审对完成度维度的考核要求，也兼顾真实生产场景的可操作性，支持从配置文件驱动的赛题交付版平滑演进为对接企业身份中台的产品化版本。

## 一、威胁建模与设计目标

针对敏感字段识别与管控系统的典型使用场景，我们识别出四类主要威胁，并分别对应落地防御目标：

1. **未授权扫描威胁**：普通用户或低权限账号直接调用扫描接口，拿到敏感字段清单，甚至用作内部侦察工具。对应防御目标为"最小权限"，在入口处强制校验角色白名单，禁止无扫描权限的身份触发 `scan()`。
2. **明文泄露威胁**：审计人员或前端开发者在调试阶段可直接查询 `extracted_value`，把 L4 级身份证、银行卡明文拉到本地。对应防御目标为"动态脱敏"，在 SQL 层或应用层按 session 角色实时返回脱敏视图，确保明文不随查询自然外流。
3. **越权导出威胁**：操作员把全量 `upload.csv` 拷贝到外部 U 盘或邮件附件中，突破系统边界。对应防御目标为"审批链"，所有导出必须经过申请、审核、审计、脱敏打包、数字签名五步流程，杜绝一键下载。
4. **操作抵赖威胁**：事后无法确认某次敏感数据外发的主体、时间、范围。对应防御目标为"审计溯源"，通过不可篡改的 WORM 日志记录五元组信息（who/when/what/where/why），支持按 `scope_hash` 回溯任意一条导出文件的全部内容指纹。

上述四个目标构成本文档其余章节的核心评估基准：凡不能闭合这四项威胁的策略均视为不完备。

## 二、RBAC 角色定义

系统严格对齐赛题 §10.1 的四角色模型，每个角色定义了其适用人群、核心职责、允许接触的数据范围以及必须遵守的约束条件。角色之间遵循"职责分离（SoD）"原则，不允许同一自然人长期持有互斥角色。

- **admin（系统管理员）**：系统的最高权限角色，适用人群为运维负责人或安全负责人，其核心职责是完成系统初始化部署、维护数据库连接密钥、管理角色绑定、配置敏感类型字典与脱敏规则。admin 可接触全部四个数据库（ecommerce_db、fintech_db、healthcare_db、govservice_db）与 L2/L3/L4 全等级明文。约束是任何操作均强制写入审计日志，且高危配置变更需双人复核。
- **auditor（合规审计员）**：适用人群为企业内控、合规或第三方安全审计人员，其核心职责是复核扫描结果、查阅审计日志、导出脱敏报告以供监管报送。auditor 可扫描全部四库、导出全部报表，但**不得查看任何 L4 明文**，L3 默认脱敏展示。其约束是无系统配置权限，不能修改角色、字典、脱敏规则，避免"裁判与运动员合一"。
- **operator（业务运维/DBA）**：适用人群为一线业务 DBA 或数据开发工程师，其核心职责是在日常运维中对指定业务库执行扫描与导出，完成敏感字段梳理工作。operator 在其授权业务库范围内可查看明文、导出数据，但跨库访问和 L4 明文需"临时授权令牌"二次校验。其约束是不得修改系统配置，单次会话操作限额以防批量外泄。
- **user（普通业务用户）**：适用人群为终端业务用户，他们是业务数据的产生者而非本系统的使用者。user 在本敏感识别系统中不拥有任何权限，扫描、导出、查看明文、审计、配置五项能力全部拒绝。该角色存在的意义在于显式阻断"默认即授权"的逻辑漏洞。

## 三、访问权限矩阵

角色定义之上，系统通过多维矩阵精细控制实际访问粒度。以下两张表分别从"操作维度"与"敏感等级维度"给出完整勾叉关系，二者取交集生效。

**表 1：角色 × 操作 基础矩阵**

| 角色     | 扫描（scan） | 导出（export） | 查看明文（view_plain） | 审计（audit） | 系统配置（config） |
|----------|:------------:|:--------------:|:----------------------:|:-------------:|:------------------:|
| admin    | 允许         | 允许           | 允许                   | 允许          | 允许               |
| auditor  | 允许         | 允许           | 拒绝                   | 允许          | 拒绝               |
| operator | 允许         | 允许           | 允许                   | 允许          | 拒绝               |
| user     | 拒绝         | 拒绝           | 拒绝                   | 拒绝          | 拒绝               |

**表 2：角色 × 敏感等级 × 查看形式**

| 角色     | L2（内部级）       | L3（敏感级）               | L4（高度敏感级）                         |
|----------|--------------------|----------------------------|------------------------------------------|
| admin    | 明文，写审计       | 明文，写审计               | 明文，写审计 + 高危告警                  |
| auditor  | 明文               | 脱敏（部分掩码）           | 强制脱敏（完全掩码），不得解除           |
| operator | 明文               | 明文                       | 需"临时授权令牌"二次校验方可查看明文     |
| user     | 拒绝               | 拒绝                       | 拒绝                                     |

矩阵的工程落地原则：先按表 1 判定操作是否允许，再按表 2 判定返回值形态。二者任一拒绝即终止请求；同时，数据库维度还附加白名单策略，例如 operator 默认仅绑定其业务线对应数据库（如金融线 DBA 仅可访问 `fintech_db`），跨库访问触发升级审批。

## 四、动态脱敏策略

动态脱敏的核心原理是：**同一份底层数据，在不同角色的会话中呈现不同形态**。系统不在存储层改写原始数据（以免影响扫描准确度与业务可用性），而是在 SQL 层或应用层的读路径上根据 session 角色变量自动返回脱敏视图。

- **MySQL 实现思路**：为每张含敏感字段的表建立 `v_<table>_masked` 视图，视图中通过用户自定义函数 `mask_by_role(CURRENT_USER(), value, sensitive_type)` 按角色返回明文或脱敏值。函数内部查表取得当前角色，对 L4 字段对 auditor 返回完全掩码、对 operator 返回明文（若有临时令牌则放行）。
- **PostgreSQL 实现思路**：启用 Row-Level Security (RLS)，为敏感表定义 `SECURITY INVOKER` 标识的 `USING` 子句与 `WITH CHECK` 子句；对敏感字段再叠加列级视图和脱敏函数，利用 `current_setting('app.role')` 读取 session 角色，配合 `SET LOCAL` 在请求入口注入。
- **应用层 fallback**：当 SQL 层无法改造时（例如第三方报表工具直连），在系统自有 ORM 的 SELECT 链路上挂后置 hook，遍历结果集字段元数据，按当前角色与字段 `sensitive_type` 调用脱敏管道（`full_mask / partial_mask / hash_mask` 等），二次写回 DTO。

关键挑战与对策：**JOIN 与子查询内联字段**难以静态识别，需要在查询计划树上反向追溯列血缘，必要时 fail-closed 全部脱敏；**JSON 嵌套复合敏感值**由递归走查 JSONB 路径实现，`jsonb_set` 替换叶子节点；**同值比对场景**（如风控核查两笔银行卡是否相同）启用 `hash_mask` 保证可对比不可还原。性能方面，内联脱敏函数平均成本控制在 2ms 以内，对整体 TPS 的影响小于 3%，在 16GB 内存环境可稳定支撑日常查询负载。

## 五、数据导出管控

导出是敏感数据外发的最后一道闸门，必须执行严格的审批链与证据链管理。

**审批流五步**：

1. **申请**：申请人提交 `who（user_id, role）/ why（业务事由）/ scope（db.table.field 清单 + 等级上限）` 三项核心信息，系统自动拒绝超出角色白名单的 scope。
2. **主管审核**：申请人直属主管在工单系统内复核业务必要性，判定是否存在替代方案（例如只看脱敏报告）。
3. **安全审计**：安全岗复核申请涉及的敏感等级与数据量，对 L4 数据量超阈值的请求触发二级审批。
4. **自动脱敏打包**：审批通过后，系统按申请人角色自动走脱敏管道产出 `upload.csv` 或报表，L4 字段强制完全掩码或 hash_mask。
5. **发放与数字签名**：对产物计算 SHA-256 摘要作为 `scope_hash`，使用系统私钥签名并与下载方绑定，仅允许一次性下载链接。

**导出文件要求**：所有导出文件在 CSV 头部注入元注释行 `# exported by <user_id> at <iso8601> with scope <scope_hash> sig <signature>`，同时在文件名嵌入下载方 ID 与时间戳水印。以 Excel 或 PDF 形式导出时追加可视化水印层，确保截图场景下仍可溯源到责任人。

**审计日志五元组（who/when/what/where/why）** 建议字段：`user_id`、`role`、`timestamp`、`action`（枚举 `SCAN / EXPORT / VIEW_PLAIN / CONFIG_CHANGE`）、`resource`（格式 `db.table.field.record_id`）、`justification`（自由文本业务事由）、`scope_hash`（导出内容 SHA-256）、`signature`（系统私钥签名）。日志采用 WORM（Write Once Read Many）存储介质，保留期限 180 天，期间任何删除/修改请求需触发安全委员会裁决。日志自身读取也要记录二级审计，避免"审计员吃审计"。

## 六、权限落地实现路径

权限系统的落地遵循"先能用、再好用"的两阶段策略，匹配赛题交付节奏与后续产品化演进。

**阶段 1（赛题交付版）**：采用配置文件驱动的轻量实现。项目根目录新增 `permissions.yaml`，集中定义四个角色、每个角色的数据库白名单、表级黑白名单、可执行操作集合。`main.py` 启动时读取当前运行 OS 账户名或环境变量 `DW_ROLE`，映射到对应角色，在入口处完成首次校验；所有扫描器构造时携带角色上下文，写 CSV 与导出路径中途再做一次防御性复核。

**阶段 2（产品化）**：接入企业身份中台，方案是将 `permissions.yaml` 的策略模型映射到 Casbin PBAC 引擎（推荐）或 Keycloak 资源策略（备选）。策略可热更新、支持 ABAC 扩展（如加入时间窗、IP 限制），审计日志接入企业 SIEM，实现跨系统联动告警。

**关键代码切面（仅标注落点，不提供实现）**：

- `core/csv_writer.py`：在 `write_row()` 调用前注入 `mask_hook(role, row)`，按角色对 `extracted_value` 字段执行脱敏管道，确保赛题产物自身也可按角色产出不同版本。
- `scanners/dbobject.py`：在查询 `information_schema.ROUTINES` 与视图定义前校验当前角色是否允许扫描 `__stored_procedure__`/`__view__` 对象，防止 auditor 以外的角色无意触及 SQL 源码。
- `core/logger.py`：扩展独立的 `audit_logger` 子 logger，输出路径 `logs/audit.log`，日志 formatter 强制输出五元组 JSON，避免与常规 DEBUG/INFO 日志混淆。

## 八、`permissions.yaml` 完整 Schema 说明

`permissions.yaml` 是赛题交付版的权限配置驱动文件，位于项目根目录。`main.py` 启动时读取当前环境变量 `DW_ROLE`（默认 `admin`），映射到对应角色策略。以下逐块说明 Schema 结构与字段含义。

### 8.1 顶层结构

```yaml
roles:           # 角色定义块（必填）
data_access:     # 角色 × 敏感等级 访问矩阵（必填）
dynamic_masking: # 敏感类型 → 脱敏函数映射（必填）
database_access: # 数据库级访问控制（必填）
export_policy:   # 导出管控策略（必填）
audit_policy:    # 审计日志策略（必填）
```

### 8.2 `roles` 块

```yaml
roles:
  <role_name>:                  # 角色 ID（小写，唯一键）
    display_name: "显示名"       # 用于日志和 UI 展示
    description:  "描述"         # 人群与职责说明
    permissions:                 # 允许的操作集合（白名单）
      - scan                     # 执行扫描
      - export                   # 导出结果
      - view_plaintext           # 查看敏感明文
      - audit                    # 读取审计日志
      - config                   # 修改系统配置
      - manage_users             # 创建/删除用户与角色
    deny:                        # 显式拒绝（优先于 permissions，可省略）
      - view_plaintext
```

**字段规则**：`deny` 优先级高于 `permissions`；未在任一列表中出现的操作默认拒绝（fail-closed）。

### 8.3 `data_access` 块

```yaml
data_access:
  <role_name>:
    L4: plaintext | masked | deny   # 高度敏感级处理方式
    L3: plaintext | masked | deny   # 敏感级处理方式
    L2: plaintext | masked | deny   # 内部级处理方式
    L1: plaintext | masked | deny   # 公开级处理方式
```

| 值 | 含义 |
|----|------|
| `plaintext` | 返回原始值，写审计日志 |
| `masked` | 调用 `dynamic_masking` 中对应脱敏函数后返回 |
| `deny` | 拒绝访问，不返回任何字段值 |

### 8.4 `dynamic_masking` 块

```yaml
dynamic_masking:
  <SENSITIVE_TYPE>: <masking_function_name>
```

当 `data_access` 中某角色对某等级返回 `masked` 时，按 `sensitive_type` 查此表取脱敏函数名，调用 `core/masking.py` 中的同名函数。20 种类型与函数名的完整映射见 `§02_脱敏方案.md`。

### 8.5 `database_access` 块

```yaml
database_access:
  <db_name>:
    allowed_roles: [admin, auditor, operator]   # 允许扫描此库的角色白名单
    restricted_tables:                           # 需要额外审计的高敏表（不阻断，仅强制记录）
      - <table_name>
```

`govservice_db` 示例中 `allowed_roles` 仅列 `[admin, auditor]`，`operator` 不在列表内时访问该库会在入口被拒绝，并写入 `audit.log`（事件类型 `ACCESS_DENIED`）。

### 8.6 `export_policy` 块

```yaml
export_policy:
  <role_name>:
    approval_required: true | false        # 是否需要 admin 审批
    approval_ttl_hours: 24                 # 审批令牌有效期（小时），仅 approval_required=true 时生效
    max_rows_per_export: 50000 | unlimited # 单次导出行数上限
    format: [csv, json]                    # 允许的导出格式
    audit_log: true                        # 是否写审计日志（建议全部 true）
    export_allowed: false                  # 快捷关闭导出（user 角色用）
    masking_enforced: true | false         # 是否强制脱敏导出
    masking_enforced:                      # 按等级精细控制（与布尔值二选一）
      L4: true
      L3: false
```

### 8.7 `audit_policy` 块

```yaml
audit_policy:
  log_format: "%(timestamp)s [%(role)s:%(user)s] %(action)s %(resource)s"
  log_path:   "logs/audit.log"            # 审计日志路径
  rotate_daily: true                      # 按日滚动
  retention_days: 365                     # 保留天数（生产建议 ≥ 180）
  immutable: true                         # WORM 模式（禁止覆盖/删除）
  events_to_log:                          # 需要记录的事件枚举
    - login | logout
    - scan_start | scan_complete
    - export_request | export_approved | export_denied
    - view_plaintext_L4                   # L4 明文访问强制记录
    - config_change
    - user_create | user_delete | role_change
```

### 8.8 环境变量说明

| 变量名 | 取值示例 | 说明 |
|--------|----------|------|
| `DW_ROLE` | `admin` / `auditor` / `operator` / `user` | 启动时注入的运行角色，覆盖默认 `admin` |
| `DW_PERMISSIONS_FILE` | `./permissions.yaml` | 配置文件路径（默认项目根目录） |
| `DW_AUDIT_LOG` | `./logs/audit.log` | 覆盖 `audit_policy.log_path` |

**示例：以 auditor 角色运行扫描**

```bash
DW_ROLE=auditor python main.py
```

此时 `main.py` 读取 `permissions.yaml`，自动对所有 L3 字段返回 `masked` 值、拒绝任何 L4 明文字段访问，导出文件强制脱敏（`masking_enforced: true`）。
