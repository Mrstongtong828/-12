# 数据库敏感字段自动识别与安全管控系统——操作手册

> 本手册面向答辩评委、助教及复现人员，按照章节顺序即可将系统从"零"搭建到产出
> `output/upload.csv`、完成本地评分、并完成脱敏与权限演示。所有命令已在 Windows 11
> + Python 3.11 + Docker Desktop 的环境下实测通过，对应的赛题机型是 i5-10400 /
> 16GB RAM，扫描全过程离线无任何外部网络调用。

---

## §一 软硬件需求

本系统严格遵循赛题"断网、无 GPU、30 分钟内扫完 4 库"三条硬约束，复现环境建议如下：

- **硬件**
  - CPU：Intel i5-10400（6C/12T，2.9GHz）或同级桌面处理器
  - 内存：16GB DDR4（实测扫描峰值约 8GB，留 8GB 给 Docker/OS）
  - 磁盘：SSD ≥ 30GB 可用空间（数据库镜像两个压缩包共 ~6GB，模型权重约 200MB）
  - 显卡：不要求，全流程 CPU 推理（`use_gpu=False`）
- **操作系统**：Windows 11（赛题标准机型）；理论可移植 Linux，但路径兼容未重点验证
- **基础软件**
  - Python 3.11.x（建议使用 64 位官方安装包，路径加入 PATH）
  - Docker Desktop ≥ 4.30（内置 docker compose v2）
  - Git（用于克隆仓库与打 tag）
  - pandoc（可选，用于 md → PDF 批量转换研究报告）
- **依赖模型**：PaddleOCR `ch_PP-OCRv4` 本地权重，已预置在 `models/paddleocr/` 下
  的三个子目录（det / rec / cls），**不得开启联网下载**
- **字符编码**：终端建议切换到 UTF-8（`chcp 65001`），避免中文日志乱码

## §二 环境准备

以下所有命令的工作目录，除特别说明外均以项目根目录
`E:\数据库敏感字段识别与安全管控系统\项目\` 为基准。

### 1. 克隆仓库

建议将仓库放在 E 盘根目录或其他全英文路径下。当前路径含中文经实测兼容，但仍推荐：

```bash
cd /e/
git clone <repo-url> 数据库敏感字段识别与安全管控系统
cd 数据库敏感字段识别与安全管控系统/项目
```

### 2. 创建并激活虚拟环境

```bash
python -m venv venv311
venv311\Scripts\activate
python -m pip install --upgrade pip
pip install -r requirements.txt
```

`requirements.txt` 已固定版本号，核心依赖包括：`pymysql==1.1.1`、`psycopg2-binary==2.9.9`、
`paddlepaddle==2.6.2`、`paddleocr==2.7.3`、`paddlenlp==2.6.1`、`opencv-python==4.6.0.66`、
`numpy==1.24.3`、`rich==13.7.1`。安装过程 paddlepaddle wheel 体积较大，国内网络建议事先
切换到清华或阿里镜像。

### 3. 验证离线依赖

PaddleOCR 默认会尝试联网下载模型，赛场必须离线，因此务必检查以下三个子目录齐全：

```
models/paddleocr/
  ├── ch_PP-OCRv4_det_infer/       (inference.pdmodel / pdiparams / pdiparams.info)
  ├── ch_PP-OCRv4_rec_infer/       (同上)
  └── ch_ppocr_mobile_v2.0_cls_infer/ (同上)
```

若任一文件缺失，启动 OCR 子进程时会抛 `FileNotFoundError`，需从离线拷贝补齐。

### 4. 启动数据库容器

```bash
cd docker数据库启动
docker compose up -d
docker ps
```

期望输出 2 个容器均为 Up 状态（每容器内部跑 2 个数据库实例）：

- `dasctf_mysql`（端口 3306，内含 `ecommerce_db`、`fintech_db`）
- `dasctf_postgres`（端口 5432，内含 `healthcare_db`、`govservice_db`）

Windows 下可直接双击 `start_all.bat`，脚本会自动唤醒 Docker Desktop、执行
`docker-compose up -d` 并拉起 DBeaver 客户端。

### 5. 健康检查

- 首选脚本：`python scripts/run_setup.py`（若仓库已提供 `check_db_health.py` 可换用）
- 手动兜底（任选其一）：
  - `mysql -h 127.0.0.1 -P 3306 -u root -prootpass123 -e "show databases;"`
  - `psql -h 127.0.0.1 -p 5432 -U postgres -c "\l"`（密码 `rootpass123`）
- **关键提醒**：必须使用 `127.0.0.1`，严禁使用 `localhost`——部分驱动在 Windows 下
  会走 Unix Socket 路径，导致 Connection refused。

## §三 扫描运行

### 主入口

```bash
venv311\Scripts\activate
python main.py
```

`main.py` 当前版本无命令行参数，所有运行配置集中在 `core/config.py`，常用项如下：

- `MAX_SCAN_MINUTES = 30`：总时限，超时后正在扫描的表会尽快收尾
- `TABLE_TIMEOUT_SECONDS = 540`：单表熔断阈值，主要保护 BLOB 表
- `DB_WORKERS = 2`：DB 级并发线程数，与 OCR slot 数对齐
- `BLOB_TABLE_MAX_ROWS = 800`：BLOB 表最多扫描行数
- `SAMPLE_THRESHOLD = 50_000` / `SAMPLE_ROWS_PER_TABLE = 6_000`：大表分层抽样策略

如需临时调参，直接修改上述常量即可，无需额外传参。

### 运行期产物

- `output/upload.csv`：**唯一提交文件**，9 列 UTF-8 无 BOM、LF 换行、RFC 4180 引号规则
- `scan_error.log`：超时 / OCR 异常 / SQL 错误的集中日志，用于事后定位失败表
- 控制台（使用 rich Console 渲染）：
  - 每个数据库一个 spinner 子任务，展示当前正在扫描的表
  - BLOB 表额外打印已扫行数 / OCR 命中数 / 单表耗时
  - 扫描结束后打印汇总表格（各形态 TP/FP/FN 不由 main.py 输出，需走 §四 评分器）

### 典型指标（i5-10400 / 16GB）

- 总耗时：24~29 分钟，卡点主要是 fintech 的两张 BLOB 表串行
- 内存峰值：6~8GB（2 个 OCR 子进程各 ~3.2GB，主进程 ~1.5GB）
- CPU 峰值：~95%（OCR 推理 + 多线程 SQL 拉取叠加）
- 产出行数：~55,962 行

## §四 本地评分与诊断

### 运行评分器

```bash
python local_scorer.py
python local_scorer.py --diff
python local_scorer.py --diff --form encoded --limit 50
python local_scorer.py --form-mismatch
```

默认读取 `output/upload.csv`（预测）与 `tests/example.csv`（1% 抽样答案）。也可显式指定：

```bash
python local_scorer.py --pred output/upload.csv --answer tests/example.csv
```

### 输出指标

评分器严格对齐赛题 §6 公式：

1. 每形态 TP / FP / FN / Precision / Recall / F1
2. 加权 F1 = Σ(权重 × F1_form)（权重：structured 10%、semi_structured 20%、
   encoded 20%、unstructured_text 25%、binary_blob 20%、db_object 5%）
3. 等级准确率（分母为"答案总行数"）
4. 综合识别得分 = 加权 F1 × 0.70 + 等级准确率 × 0.30
5. FP 样本前 20 条（定位过宽正则误报）
6. FN 样本前 20 条（定位召回缺口）

### diff_report.txt 解读

- `[FP]`：预测命中但答案中没有（常见：正则太宽、字段名误判）
- `[FN]`：答案有但预测缺失（常见：编码轮数不够、OCR 漏识别、姓名黑名单太严）
- `[FORM_MISMATCH]`：值对但 `data_form` 标错（典型：把 `encoded` 标成 `structured`）

## §五 常见故障排查

| 现象 | 可能原因 | 处置建议 |
|------|---------|---------|
| `[OCR#x] slot 永久停用` | PaddleOCR primitive 创建失败（多半是裸 RIFF 非 WEBP 格式或 MKL 线程冲突） | 重启扫描；长期改 `scanners/blob.py` 加格式白名单过滤 |
| 数据库 `connection refused` | 容器未 Up / 端口被占用 / 使用了 `localhost` | `docker ps` 确认 4 库 Up；`netstat -ano` 查 3306/5432；严格使用 `127.0.0.1` |
| `表 X 超时(540s)` | BLOB 表图片数量超出预算 | 已是上限，只能降 `BLOB_TABLE_MAX_ROWS` 或切 Plan B 表级并行 |
| 内存 OOM / 进程被杀 | OCR 子进程叠加超 16GB | 降 `OCR_POOL_SIZE` 到 1，或关闭 Docker 以外的桌面软件释放 RAM |
| `upload.csv` 写入失败 | 目录无权限 / Excel 占用了文件 | 检查 `output/` 权限；**确认 CSV 未在 Excel 中打开** |
| 总耗时超 30 min | 多半卡在 fintech BLOB 表 | 观察控制台定位瓶颈，再调低 `BLOB_TABLE_MAX_ROWS` 下限 |
| 中文日志显示 `???` | 终端非 UTF-8 | `chcp 65001` 切换终端编码，或使用 Windows Terminal |
| PaddleOCR 尝试联网 | `download_enabled=True` | 确认启动时 `use_gpu=False` 且所有 model_dir 指向本地目录 |

## §六 演示视频分镜

建议 8 段式剪辑，总时长控制在 5~8 分钟以内，每段按下方要点拍摄。

1. **片头 + 作品简介**（10~15s）：作品名、队伍名、赛题定位一句话说明。
2. **系统架构图解说**（30s）：展示分层架构（DB 连接层 / Dispatcher / 六形态 Scanner /
   CSV Writer / OCR 池），强调"六形态全覆盖 + 离线 OCR + 并发熔断"。
3. **环境启动演示**（30s）：录 `cd docker数据库启动` → `docker compose up -d` →
   `docker ps` 展示 2 容器 Up，画外音强调"离线、127.0.0.1"。
4. **扫描运行实录**（30s，可 10× 加速）：`python main.py`，画面展示 rich 进度条，
   重点框出 BLOB 表行数计数与两个 OCR slot 的并发状态。
5. **输出展示**（30s）：VSCode 打开 `output/upload.csv`，随机抽 3 行高亮
   `(db_type, db_name, table_name, field_name, record_id, data_form,
   sensitive_type, sensitive_level, extracted_value)` 9 列，强调无 BOM、LF、RFC 4180。
6. **本地评分**（40s）：`python local_scorer.py`，逐一念出六形态 F1 与加权 F1，
   画面框住"综合识别得分"一行。
7. **脱敏 Demo**（30s）：`python -m core.masking`，展示 PHONE_NUMBER /
   ID_CARD / BANK_CARD / EMAIL / PASSWORD_OR_SECRET 五个典型类型的前后对比。
8. **权限 Demo**（30s）：用 admin / auditor / operator / user 四种角色登录同一视图，
   展示返回明文 vs 动态脱敏 vs 拒绝访问的差异，呼应 RBAC 策略章节。
